# Home Icon Studio

## プロジェクト概要

Home Icon Studio は、ユーザーが用意したホーム画面の録画から、表示されているアプリ名候補を抽出し、ショートカット等で使えるシンプルなカスタムアイコン画像の作成を補助する Web アプリケーションです。

元の構想は iOS アプリでしたが、このプロジェクトでは App Store 配布を前提にせず、個人利用向けの Web アプリとして実装します。ブラウザで操作できる UI を中心にし、動画解析や OCR のような重い処理だけをサーバー側の Python ワーカーで補助するハイブリッド構成を採用します。

## 解決したい課題

ホーム画面のカスタムアイコンをまとめて作る作業は、対象アプリの洗い出し、アイコン生成、書き出し、ショートカット設定の確認に手間がかかります。このプロジェクトは、ユーザーが提供した録画をもとに見えているアプリ名を推定し、確認済みの候補に対してオリジナルのアイコン画像を生成することで、その作業を短縮します。

このアプリは、端末内のインストール済みアプリ一覧を取得するツールではありません。録画に見えている文字から候補を作り、最終的な確認と修正はユーザーが行います。

## MVP で提供すること

- ホーム画面録画のアップロード
- 解析ジョブの開始と進捗確認
- 動画からの代表フレーム抽出
- OCR によるアプリ名候補の抽出
- 候補の確認、編集、削除、追加
- シンプルな PNG アイコン生成
- 生成済みアイコンのダウンロード
- 一時ファイルと解析結果の削除

## MVP で提供しないこと

- iOS ネイティブアプリとしての実装
- App Store 配布
- インストール済みアプリ一覧の取得
- ホーム画面への自動配置
- マルチユーザー対応
- 認証基盤
- クラウド OCR
- 外部ストレージ連携
- 本格的なジョブキューや DB

## 技術選定

### フロントエンド

- `Next.js`
- `React`
- `TypeScript`
- `Tailwind CSS`

画面、アップロード、レビュー UI、テーマ選択、ダウンロード導線を 1 つの Web アプリとして実装します。個人利用の初期開発では、フロントエンドと軽量 API を同じ Next.js アプリに置く方針です。

### サーバー層

- `Next.js Route Handlers`
- 必要に応じて `Server Actions`

アップロード受付、解析ジョブの起動、結果取得、生成済みアセットの配布を担当します。初期段階では DB やキュー基盤を導入せず、ローカルファイルと JSON メタデータで扱います。

### 解析ワーカー

- `Python`
- `ffmpeg`
- `PaddleOCR` または `EasyOCR`
- `Pillow`

動画のフレーム抽出、安定フレーム選定、OCR、候補生成、PNG アイコン生成を担当します。OCR エンジンは日本語対応を重視し、実装時に `PaddleOCR` と `EasyOCR` のどちらかを選びます。

### データ保存

初期構成ではローカルファイル保存を採用します。

- `uploads/`: 取り込んだ録画
- `tmp/`: 解析中の一時ファイル
- `artifacts/`: OCR 結果、生成アイコン、書き出し用ファイル
- JSON: 解析結果や候補一覧などの軽量メタデータ

## 想定アーキテクチャ

```text
Browser
  - 動画選択
  - 候補レビュー
  - テーマ選択
  - ダウンロード

Next.js
  - 画面提供
  - アップロード受付
  - 解析ジョブ起動
  - 結果取得 API
  - アセット配布

Python Worker
  - ffmpeg によるフレーム抽出
  - 安定フレーム選定
  - OCR
  - アプリ名正規化
  - PNG アイコン生成

Local Storage
  - uploads/
  - tmp/
  - artifacts/
  - JSON metadata
```

## 主要ユーザーフロー

1. ユーザーがホーム画面を録画する。
2. Web アプリで録画ファイルを選択する。
3. アプリが録画をアップロードし、解析ジョブを開始する。
4. Python ワーカーが代表フレームを抽出し、OCR を実行する。
5. アプリが検出されたアプリ名候補を表示する。
6. ユーザーが候補を確認、編集、削除、追加する。
7. ユーザーがテーマを選び、アイコンを生成する。
8. ユーザーが生成済みアイコンをダウンロードする。
9. ユーザーが不要になった録画や解析結果を削除する。

## ディレクトリ方針

実装開始時は、次の構成を基本にします。

```text
app/
  api/
  page.tsx
  layout.tsx

components/
  upload/
  analysis/
  review/
  icon-studio/

lib/
  jobs/
  storage/
  schemas/

worker/
  frame_extraction/
  ocr/
  matching/
  icon_generation/

data/
  app-name-dictionary.json
  default-icon-themes.json

uploads/
tmp/
artifacts/
```

`uploads/`, `tmp/`, `artifacts/` は個人利用のローカル保存領域です。将来クラウドや DB を導入する場合も、まずはこの責務分割を保ったまま置き換えます。

## 開発方針

- まずは個人利用で動く MVP を優先する。
- ブラウザ UI と軽量 API は Next.js にまとめる。
- 重い解析処理は Python ワーカーに分離する。
- 解析結果はユーザー確認を前提にし、OCR 結果を確定情報として扱わない。
- サードパーティのロゴや既存アプリアイコンをコピーしない。
- 生成アイコンは頭文字、抽象図形、単純な背景などのオリジナル表現を基本にする。
- 録画、解析結果、生成物を削除できる導線を用意する。
- 未使用の認証、DB、キュー、クラウド連携は追加しない。

## 実装優先順位

1. Next.js の最小アプリ構成を作る。
2. 録画ファイルのアップロードを実装する。
3. ローカル保存領域 `uploads/`, `tmp/`, `artifacts/` を扱うユーティリティを作る。
4. Python ワーカーを呼び出す最小ジョブ導線を作る。
5. `ffmpeg` による代表フレーム抽出を実装する。
6. OCR の概念実証を実装する。
7. アプリ名候補の正規化とレビュー UI を作る。
8. シンプルな PNG アイコン生成を実装する。
9. 生成物のダウンロードを実装する。
10. 録画、解析結果、生成物の削除を実装する。

## データとファイルの扱い

- アップロードされた録画は `uploads/` に保存する。
- 解析途中のフレームや一時ファイルは `tmp/` に保存する。
- OCR 結果、候補 JSON、生成アイコンは `artifacts/` に保存する。
- 解析結果 JSON には、元の OCR 文字列、正規化後の表示名、信頼度、ページまたはフレーム情報を含める。
- 個人利用の初期段階では DB を使わない。
- 削除操作では対象ジョブに紐づく録画、一時ファイル、生成物を消せるようにする。

## MVP 完了条件

- ユーザーが Web UI からホーム画面録画をアップロードできる。
- Python ワーカーが録画から代表フレームを抽出できる。
- OCR により表示テキスト候補を取得できる。
- ユーザーが候補を確認、編集、削除、追加できる。
- 確認済み候補からオリジナルの PNG アイコンを生成できる。
- ユーザーが生成済みアイコンをダウンロードできる。
- 録画、解析結果、生成物を削除できる。
- 実装が Web アプリ前提の技術選定に沿っている。
