# Web 版プロジェクト文書整備 Design

## 概要

`CODEX_PROJECT_BRIEF.md` を基に、iOS ネイティブ前提の説明をそのまま流用せず、個人利用向けのハイブリッド Web アプリとして再解釈した文書を整備する。作成対象は人間開発者向けの `README.md` と、AI エージェント向けの `CODEX.md` である。

この設計の目的は、まだ実装コードが存在しない段階で、今後の開発判断が iOS ネイティブ前提に引き戻されたり、過剰な基盤導入に流れたりしないよう、Web 前提の技術選定と制約を明文化することにある。

## 前提条件

- 対象は App Store 配布の iOS アプリではなく、個人利用を前提とした Web アプリである。
- 実行形態はハイブリッド構成とし、ブラウザ UI とサーバー補助処理を組み合わせる。
- プライバシー方針は「実装優先」で決めるが、初期設計としてはローカル保存中心・個人利用前提を採用する。
- マルチユーザー対応、認証基盤、クラウド前提アーキテクチャは MVP スコープ外とする。

## 目的

### README.md

`README.md` は人間開発者向けの入口文書とする。以下を短時間で把握できる内容にする。

- 何を作るプロジェクトか
- 何を MVP として提供するか
- なぜその技術選定なのか
- どういう責務分割で実装を始めるのか

### CODEX.md

`CODEX.md` は Codex や同種の AI エージェント向け実装ガイドとする。以下を明文化する。

- このプロジェクトを Web アプリとして扱うこと
- 採用技術と非採用技術
- フロントエンド、Next.js、Python ワーカーの責務境界
- 個人利用前提ゆえに避けるべき過剰設計
- 実装時の禁止事項と優先順位

## 技術選定

### 推奨アプローチ

初期構成は `Next.js` を中心とした単一リポジトリ構成とする。UI と軽量 API を同居させ、重い解析処理のみ Python ワーカーへ委譲する。

この案を採用する理由は以下の通り。

- 個人利用では運用負荷が低い
- Web UI と API 導線を 1 つのアプリケーションで整理しやすい
- Python 側の OCR / 動画処理だけ独立させれば、将来的な差し替えも容易
- 初期段階でフロントエンドとバックエンドを完全分離するより実装判断が単純

### 採用技術

- フロントエンド: `Next.js` / `React` / `TypeScript` / `Tailwind CSS`
- Web サーバー層: `Next.js Route Handlers` または `Server Actions`
- 重い解析処理: `Python` ワーカー
- 動画処理: `ffmpeg`
- OCR: `PaddleOCR` または `EasyOCR`
- 画像生成: サーバー側は `Pillow`、プレビュー用途は `Canvas` または SVG
- データ保存: `uploads/`, `artifacts/`, `tmp/` を中心としたローカルファイル保存 + `JSON`

### 非採用方針

初期段階では以下を採用しない。

- 認証基盤
- DB
- キュー基盤
- 外部オブジェクトストレージ
- クラウド OCR
- マルチユーザー対応
- iOS ネイティブ API 前提の設計

## 責務分割

### ブラウザ

- 動画ファイルの選択
- 解析開始操作
- 候補レビュー
- アイコンテーマ選択
- 生成結果確認
- ダウンロード操作

### Next.js

- 画面提供
- アップロード受付
- 解析ジョブ起動
- 結果取得 API
- 生成済みアセットの配布

### Python ワーカー

- フレーム抽出
- 安定フレーム選定
- OCR
- アプリ名正規化
- 候補生成
- PNG アイコン生成

## 文体と記述方針

### README.md

- 日本語のみで記述する
- 人間開発者向けの案内文書として書く
- 背景説明よりも、何を作るかとどう始めるかを優先する
- 未確定のセットアップ手順や未実装コマンドは書かない

### CODEX.md

- 日本語のみで記述する
- AI エージェント向けに命令形と制約中心で書く
- 裁量の余地を減らすため、採用技術・禁止事項・優先順位を明示する
- 「Web 前提」「個人利用前提」「ローカル保存中心」「重い解析のみ Python ワーカー補助」を繰り返し固定する

## スコープ外

今回の作業では以下は行わない。

- 実アプリコードの生成
- セットアップコマンドの確定
- CI/CD やデプロイ手順の決定
- OCR エンジンの最終比較検証
- `CODEX_PROJECT_BRIEF.md` 自体の全面的な Web 版書き換え

## 完了条件

この設計に基づく文書作業の完了条件は以下とする。

- `README.md` が人間開発者向けの入口文書として成立している
- `CODEX.md` が AI エージェント向け実装ガイドとして成立している
- 両文書が Web アプリ前提で整合している
- 技術選定、責務分割、禁止事項に矛盾がない
- iOS ネイティブ前提の判断を新規文書へ持ち込まない
